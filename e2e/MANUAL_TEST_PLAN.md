> 실행 로그/발견사항은 [MANUAL_TEST_PROGRESS.md](./MANUAL_TEST_PROGRESS.md)에 시간순으로 기록.

## 1. 테스트 방식 (왜 이렇게 하는가)

### 인증 구조상의 제약
- 세션은 `rp_session` **httpOnly 쿠키**(`src/lib/session.ts`, path `/`)로 동작.
- **같은 Chrome 프로필의 모든 탭은 이 쿠키를 공유** → 한 프로필 탭 3개로 서로 다른 3명 동시 로그인 불가(마지막 로그인이 덮어씀).
- **시크릿 창**도 모든 시크릿 탭이 단일 쿠키 저장소 공유 → "추가 세션 1개"만 제공(일반+시크릿 = 최대 2명). 게임 최소 인원 3명에는 부족.
- 추가로 `active_sid`(users 테이블)로 "한 계정 1세션"을 강제 → 같은 계정 재로그인 시 이전 세션 무효화(SessionGuard).
- 결론: 3명 동시 세션 = **쿠키 저장소 3개 필요**.

### 채택한 하이브리드 방식
- **방장(시오, `mathasdf0@gmail.com`)**: 사용자의 **실제 Chrome 탭**에서 Claude-in-Chrome MCP로 직접 클릭·조작. → 이전/다음 버튼, 채팅, 레이아웃을 실제 화면에서 육안 검증.
- **피어 2명(제이크 `kde2800623@gmail.com`, 루디 `sunwoo005@gmail.com`)**: 독립 쿠키 저장소를 갖는 **별도 실제 브라우저 세션**(Playwright chromium context)으로 동시 구동. → 입장/준비/작성/채팅을 실제로 발생시킴. 스크립트: [peers_runner.mjs](./peers_runner.mjs).
- 방장이 채널을 먼저 구독해야 피어들의 ready/채팅 broadcast가 도달하므로 **방장 입장 → 피어 기동** 순서.
- (대안) 진짜 3창이 필요하면 Chrome 프로필 3개에 각각 확장 로그인. 설정 부담 커서 보류.

### 공통 셋업
- dev 서버: `npm run dev` (localhost:3000). Preview 도구 또는 직접 기동. 포트 점유 시 `lsof -ti:3000` 정리.
- 테스트 방: **7번 방**. DB 초기화: `resetTestRoom(7)` 로직 = `reset_room.mjs`(임시 스크립트, helpers.ts 참고: room_members/assignments/messages 삭제 후 rooms state=lobby).
- 실 Supabase + Realtime 사용(`.env.local` 필요).
- 동기화 로직: 준비/모드/채팅/공개이동 모두 `room-{roomId}` 채널 **broadcast**. 채팅은 영속 X(구독 중일 때만 수신).

### 검증 도구
- 실제 Chrome(방장): `mcp__Claude_in_Chrome__*` (navigate/find/read_page/computer/form_input/read_console_messages/read_network_requests).
- 레이아웃: 스크린샷 + `resize_window`(mobile 375 / tablet 768 / desktop 1280).
- 에러: read_console_messages(onlyErrors), read_network_requests(실패 필터).

## 3. 진행 순서(런북)
1. dev 서버 확인(3000), `resetTestRoom(7)`.
2. 방장(실제 Chrome): 로그인(시오) → `/rooms` 레이아웃 점검(B) → 7번 방 입장 → lobby 점검(C 일부).
3. `node e2e/peers_runner.mjs` 백그라운드 기동 → `PEERS_READY` 로그 대기(피어 입장·준비·로비채팅).
4. 방장 화면에서 C2/C3/C4/C5/C10(채팅 수신) 확인.
5. 방장 "게임 시작하기"(C6) → 작성(D) 점검. 피어는 스크립트가 자동 제출(`PEERS_WROTE`).
6. 방장도 2개 답변 제출 → 전원완료 자동 공개(D4) → 공개(E) 점검: **이전/다음·연타·바로가기·익명/실명·채팅**.
7. 마지막 장 "다음" → 종료(F) 점검: 결과 모달·이전/다음 답변·다시하기.
8. 반응형(G) 각 단계 resize 점검.
9. 정리: `resetTestRoom(7)`, 피어 스크립트 종료.

## 4. 알려진 제약 / 메모
- 채팅은 broadcast-only(비영속): 방장이 구독 중일 때 전송된 메시지만 수신.
- 피어는 헤드리스 실제 브라우저(별 쿠키)라 "사용자 화면에 보이는 탭"은 아님 — 방장 탭만 육안 노출.
- A4(active_sid)·B4(정원)·B5(타방 점유)·H2(승계) 등은 추가 계정/상태 셋업이 필요 → 시간 허용 시 보강.
